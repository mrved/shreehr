```
describe('Attendance Workflows')
  describe('Employee Check-in/Check-out')
    test.use({ userRole: 'employee' })

    - 'should display attendance page for employee'
      -> navigate to /employee/attendance, assert calendar or attendance view visible
    - 'should show check-in button when not checked in'
      -> navigate to /employee/attendance, assert check-in button visible, check-out button not visible (or disabled)
    - 'should check in successfully'
      -> click check-in, assert status changes to show checked-in state, assert check-out button appears
    - 'should check out and show work hours'
      -> (after check-in) click check-out, assert work hours displayed, assert status updated

    NOTE: Check-in/out tests should be in serial mode since check-out depends on check-in.
    Use `test.describe.configure({ mode: 'serial' })`.

  describe('Admin Team Attendance')
    test.use({ userRole: 'admin' })

    - 'should display team attendance page'
      -> navigate to /dashboard/attendance, assert page heading visible, assert team data table/list visible
    - 'should show attendance records for team members'
      -> navigate to team attendance page, assert at least one row visible

  describe('Attendance Lock')
    test.use({ userRole: 'admin' })

    - 'should display attendance lock management page'
      -> navigate to /dashboard/attendance/lock, assert lock management controls visible
    - 'should lock attendance for a period'
      -> select a month/year, click lock, assert lock status changes
    - 'should show locked status for previously locked period'
      -> after locking, verify the period shows as locked

    NOTE: Lock tests may modify state. Use serial mode and consider cleanup.
```

- Attendance page may need time for calendar to render -- use waitFor on calendar element
- Lock tests should use a past month (not current) to avoid interfering with check-in/out tests
